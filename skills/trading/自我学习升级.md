## 今日问题反思 (2026-03-16)

### ❌ 问题 1: 数据解析错误

**问题描述**:
- 把换手率 (22.25%/24.98%) 当成涨跌幅返回
- 导致多次报告"涨停 20%+"(A 股涨跌停限制 10%)
- 用户多次质疑后才纠正

**根本原因**:
1. MCP API 返回数据中 `change_pct` 字段实际是换手率
2. 没有交叉验证数据合理性
3. 没有检查 A 股涨跌停限制 (10%)

**解决方案**:
```
1. 数据获取后必须进行合理性校验:
   - 涨跌幅 >10% → 数据错误，重新获取
   - 涨跌幅 < -10% → 数据错误，重新获取
   
2. 多数据源交叉验证:
   - MCP API + 雪球浏览器对比
   - 差异>5% → 标记可疑，人工确认
   
3. 添加数据校验层:
   - 涨跌幅范围检查 (-10% to +10%)
   - 换手率范围检查 (0% to 50%)
   - 成交额合理性检查
```

**教训记录**:
> ⚠️ **永远不要盲目相信单一数据源！**
> ⚠️ **必须进行合理性校验！**
> ⚠️ **A 股涨跌停限制 10% 是硬约束！**

### ❌ 问题 2: 收盘后继续盘中监控

**问题描述**:
- 15:00 收盘后，用户问"收盘了监控啥"
- 才发现没有自动禁用盘中监控

**根本原因**:
1. 盘中监控 cron 没有时间范围限制
2. 没有添加收盘自动暂停逻辑

**解决方案**:
```
1. 修改 cron 表达式:
   - 原：*/1 9-11,13-15 * * 1-5 (每 1 分钟)
   - 新：*/1 9-10,13-14 * * 1-5 (15:00 前停止)
   
2. 添加收盘自动暂停:
   - 15:00 自动禁用盘中监控 cron
   - 次日 9:25 自动启用
   
3. 添加状态检查:
   - 每次监控前检查当前时间
   - >15:00 → 跳过并提醒用户
```

**教训记录**:
> ⚠️ **盘中监控必须有明确的时间边界！**
> ⚠️ **收盘后自动暂停，次日自动恢复！**

---

### ❌ 问题 3: 数据源优先级执行不严格

**问题描述**:
- 技能文档规定"雪球优先"
- 实际执行中仍在使用 MCP API
- 导致数据错误

**根本原因**:
1. 没有严格执行数据源优先级
2. MCP API 调用更简单，习惯性使用
3. 缺乏数据源质量监控

**解决方案**:
```
1. 强制数据源优先级:
   - 实时数据：雪球浏览器 (必须)
   - 历史数据：MCP API (可选)
   
2. 添加数据源质量评分:
   - 每次数据获取后评分
   - 错误率>10% → 降级/弃用
   
3. 数据源切换日志:
   - 记录每次数据源选择
   - 记录错误原因
```

**教训记录**:
> ⚠️ **严格执行数据源优先级！**
> ⚠️ **实时数据必须用雪球！**

## 自我进化机制

### 每日反思 (23:00)

**流程**:
1. 回顾今日所有操作
2. 识别错误和问题
3. 分析根本原因
4. 制定解决方案
5. 更新技能文档
6. 记录教训

**输出**:
- `memory/自我进化/YYYY-MM-DD.md`
- 更新相关技能文档

### 每周总结 (周日 22:00)

**流程**:
1. 统计本周错误数量
2. 分析错误类型分布
3. 评估改进效果
4. 制定下周改进计划

**输出**:
- `memory/自我进化/周总结_YYYY-MM-DD.md`

## 错误分类

### A 类错误 (严重)
- 数据错误导致用户决策失误
- 违反交易规则 (如 T+1)
- 未执行止损止盈

**处理**: 立即修复 + 添加防护层

### B 类错误 (中等)
- 数据延迟/不准确
- 监控时间错误
- 报告格式问题

**处理**: 24 小时内修复

### C 类错误 (轻微)
- 错别字/格式问题
- 非关键信息错误

**处理**: 48 小时内修复
